# 맨투맨 (M2M) — AI 진로 멘토링 서비스

> **"직무 정보 접근성의 불평등은 개인의 노력 부족이 아니라, 연결·식별·발화·자산화라는 네 단계의 구조적 막힘에서 비롯된다."**

막연한 진로 고민을 구조화하고, 기존 멘토 답변 자산으로 답변 가능한지 검증한 뒤, 필요한 경우 적합한 현직자 멘토에게 연결하며, 새 멘토 답변을 다시 자산화하는 **AI 진로 멘토링 서비스 구현 프로젝트**입니다.

4개의 판단형 Agent로 구성된 MVP입니다.

---

## 📌 문제 정의

### 채용 구조의 전환이 만든 정보 비대칭

채용시장이 공채 중심에서 직무 중심 수시채용으로 전환되며(89.8%, KDI 2025), 기업은 직무 이해도와 실무 경험을 중심으로 선별하는 구조로 변화했다. 이 변화는 지원자에게 새로운 요구를 부과했다 — 특정 직무가 실제로 무엇을 하는지, 어떤 준비가 필요한지를 **스스로 탐색하고 이해해야 한다**는 것이다.

그러나 그 탐색에 필요한 현직자 경험은 개인 네트워크에 따라 접근성이 크게 달라진다. 대학생의 46.9%가 진로를 결정하지 못한 상태이며(잡코리아 2021), 한국직업능력연구원 대학 진로교육 현황조사(2024)에서도 구체적 취업정보 수집 어려움과 실무경험 부족이 반복 보고된다.

이것은 **정보의 절대량 문제가 아니라 접근 경로의 불평등 문제다.**

### 사회적 자본의 분배 불평등

동일한 대학생이라도 가족·선배·지인 네트워크를 통해 현직자와 연결되는 경우와 그렇지 못한 경우 사이에는 직무 정보 획득 기회에 상당한 차이가 존재한다. 이는 개인의 노력 차이가 아니라 **사회적 자본(social capital)의 분배 불평등**이다. 부르디외(Bourdieu)의 자본 이론으로 보면, 경제적 자본과 달리 사회적 자본은 가시화되지 않기 때문에 격차가 구조적으로 재생산된다.

직무 중심 채용은 학생에게 더 많은 자기주도적 탐색을 요구하지만, 그 탐색에 필요한 자원은 균등하게 분배되어 있지 않다. 따라서 맨투맨은 단순한 정보 제공 서비스가 아니라, **현직자 경험으로 가는 경로를 AI가 매개하는 구조**를 목표로 한다.

### 구조적 차단의 4단계

현직자 경험에 도달하는 과정은 다음 네 단계에서 반복적으로 막힌다.

| 단계 | 막히는 지점 | 기존 서비스의 한계 | 맨투맨의 대응 |
|------|------------|-------------------|-------------|
| ① **질문 설계** | 고민을 구체적 질문으로 정리하지 못한 채 대화 시작 | 막연한 질문이 정제 없이 멘토에게 전달 → 양쪽 비효율 | Agent 1: 멀티턴 대화로 멘티를 프로파일링하며 질문을 구조화. 단순 텍스트 정제가 아니라 막힌 지점의 성격·위험 신호·검색 전략 힌트까지 추출 |
| ② **멘토 필요 여부 판단** | 기존 답변으로 충분한지, 멘토가 필요한지 알 수 없음 | 모든 질문이 멘토에게 전달 → 멘토 부담 과중, 대기 발생 | Agent 2: 임베딩 검색 후 5기준 검증 → 직접 답변 / 부분 답변 / 멘토 연결 3방향 분기 |
| ③ **멘토 식별** | 어떤 기준으로 적합한 현직자를 고를지 알 수 없음 | 멘티에게 "찾는 책임" 부과 — 식별 역량 자체가 정보 자본 | Agent 3: 직무 유사도뿐 아니라 멘티의 막힌 지점·전환 방향·전환 경험 보유 여부 기반 Top-3 매칭 |
| ④ **경험의 자산화** | 1:1 답변은 한 멘티에게만 전달되고 사라짐 | 매년 같은 질문 반복, 멘토 부담 누적 | Agent 4: 품질 판단 후 임베딩·태그와 함께 저장 → Agent 2 검색 대상으로 즉시 편입 |

## 🤖 왜 Multi-Agent Architecture인가

처음에는 LLM이 진로 질문에 직접 답하는 챗봇 형태도 가능하다고 생각했다. 그러나 실제 멘토링 질문 50건을 검토하면서 문제는 단순 답변 생성이 아니라는 점을 확인했다.

하나의 질문 안에 정보 탐색, 개인 상황 판단, 자료 피드백 요청, 멘토 경험 요청이 섞여 있었다. 단일 LLM 호출로는 **질문이 충분히 정제됐는지, 기존 답변으로 답해도 되는지, 어떤 멘토가 적합한지, 답변을 어떻게 재사용할지**를 분리해서 판단할 수 없다.

각 판단을 독립적인 에이전트로 분리하면 두 가지 이점이 생긴다. 첫째, 각 에이전트의 프롬프트와 판단 로직을 독립적으로 개선할 수 있다. 둘째, 중간 결과(Agent 1의 구조화 출력)가 명시적인 데이터로 저장되므로 어느 단계에서 오류가 발생했는지 추적할 수 있다.

맨투맨의 목표는 LLM이 현직자를 대체하는 것이 아니다. **멘토링 자원이 필요한 질문을 더 정확히 선별하고, 반복되는 질문은 자산화된 답변으로 효율적으로 처리하는 것**이다. 각 에이전트는 그 목표를 위해 서로 다른 판단 책임을 분담한다.

## 🧩 Agent 설계

### Agent 1 — 멘티 프로파일링 + 질문 구조화 (`agents/question_refine_agent.py`)

> 막연한 고민을 멘토가 답할 수 있는 질문으로 만드는 것이 목표지만, 그 과정에서 멘티를 프로파일링하고 구조화한 맥락을 Agent 2·3에 전달하는 것이 실제 핵심 역할이다.

**대화 루프 설계** — 멀티턴으로 현재 상태·관심 직무·보유 경험·제약 조건·알고 싶은 내용 5개 항목을 수집한다. 충분성 판단을 단순한 "필드 채움 여부"로 처리하지 않고 **항목별 가중합 점수(≥ 0.75)** 방식으로 설계했다. "알고 싶은 내용"이 막연하면 나머지가 채워져도 정제 단계로 넘어가지 않는다.

**이중 프롬프트 구조** — 정제 단계는 REFINEMENT_PROMPT 하나로 끝내지 않는다. 정제 후 CHECK_PROMPT로 자기검증을 한 번 더 거쳐 출력을 확정한다. 이 self-refine 패턴은 구조화 출력의 일관성을 높이기 위해 도입했다.

**구조화 출력** — 최종 출력은 정제된 텍스트 질문 하나가 아니다. 복합 질문의 독립 단위 분리, 막힌 지점 분류(9개 표준 라벨), 위험 신호 플래그, 검색·매칭 전략 힌트, 도메인·직무·커리어 단계 태그 전체가 `question_sessions.json`에 저장돼 Agent 2·3의 판단 입력으로 전달된다. 9개 라벨과 hard-case 규칙은 실제 잇다 Q&A 50건 분석에서 귀납적으로 도출했다.

**Hard-case 규칙 내장** — 실제 질문 패턴에서 반복된 까다로운 케이스를 프롬프트가 아닌 코드 레벨 규칙으로 처리한다. 자소서·포트폴리오 피드백 요청은 검색 없이 즉시 멘토 연결, "최근 트렌드" 질문은 Agent 2 검증 시 최신성 가중치 상향, 개인정보가 포함된 질문은 PII 패턴 감지 후 원본을 제거한 safe_context로 대체한다.

**프롬프트 엔지니어링 설계** — 기법을 목적에 따라 분리해 적용했다. 대화 생성(SYSTEM_PROMPT)과 충분성 평가(CHECK_SYSTEM)를 완전히 다른 역할 컨텍스트로 실행해 평가자가 대화 흐름에 오염되지 않도록 했다. SYSTEM_PROMPT에 8개 대화 예시를, REFINEMENT_PROMPT에 5개 정제 예시를 **few-shot**으로 삽입해 출력 형식과 판단 방향을 고정했다. 정제 후 REFINE_CHECK_PROMPT가 13개 기준으로 출력을 검토하고, 실패 항목이 있으면 fix_instructions와 함께 재생성하는 **Self-Refine** 루프를 1회 적용한다. Hard-case Rules는 "전환 조건 3가지를 모두 충족해야 직무전환으로 분류"처럼 순서 있는 조건 체크를 강제하는 **CoT** 구조다. 대화 temperature=0.7, 정제 temperature=0.3, 평가 temperature=0으로 단계별로 달리 설정해 창의성과 일관성을 목적에 맞게 조절했다.

### Agent 2 — 검색·검증·분기 (`agents/search_verify_agent.py`)

> 단순 RAG가 아니다. 검색보다 분기 판단이 핵심이다 — 기존 자산으로 답할 수 있는 질문과 멘토가 반드시 필요한 질문을 정확히 가르는 것이 이 에이전트의 존재 이유다.

**선처리 → 전략 선택 → 검색 → 검증 → 분기** 순서로 작동한다. 자료 검토 플래그가 감지된 질문은 검색 자체를 생략하고 즉시 멘토로 넘긴다. 나머지는 Agent 1이 판단한 질문 성격에 따라 `search_first`(기존 자산 우선)와 `mentor_first`(개인 맥락 우선) 두 전략 중 하나를 선택하고 임베딩 검색으로 상위 5개 후보를 추출한다.

**5기준 LLM 검증** — 코사인 유사도만으로 답변 품질을 판단하지 않는다. 관련성·근거 충분성·최신성·상황 적합성·개인정보 안전 5개 기준으로 LLM이 각 후보를 0.0~1.0으로 채점하고 가중합 점수를 산출한다. 최신성 민감 질문은 최신성 기준의 가중치를 자동으로 높인다.

**전략별 차등 임계값** — search_first와 mentor_first는 질문 성격이 다르기 때문에 직접 답변 허용 기준을 다르게 설정했다. 개인 맥락이 강한 mentor_first 질문에 더 높은 기준(0.75)을 적용해 오답 위험을 줄인다. 점수 구간에 따라 AI 직접 답변 / 부분 답변 + 멘토 권유 / Agent 3 이관 3방향으로 분기한다.

**stale_but_useful 처리** — 시의성 판단을 통과/실패 이분법이 아닌 3단계로 설계했다. "오래됐지만 여전히 유용한" 답변은 폐기하지 않고 `requires_latest_check` 플래그와 함께 제한적으로 활용한다.

**프롬프트 엔지니어링 설계** — LLM이 해야 할 의미 판단과 코드가 해야 할 수치 계산을 명확히 분리했다. LLM은 관련성·충분성·상황 적합성·개인정보 안전성의 의미적 판단만 담당하고, 가중합산·임계값 비교·분기 결정은 코드가 실행한다. VERIFY_PROMPT의 각 채점 기준을 0.9~1.0 / 0.7~0.8 / 0.5~0.6 구간별 구체적인 판단 근거와 함께 명시해(**criteria-based prompting**) LLM의 점수 편향을 줄였다. 직무 전환 질문에 대해서는 "source→target 전환 논리를 설명하지 않으면 situation_fit을 낮게 주라"는 기준을 VERIFY_PROMPT 안에 직접 내장했다. FAITHFULNESS_PROMPT로 생성된 답변의 근거 충실성·사실 정확성·개인정보 노출을 자기검증하며, 2회 미통과 시 멘토 연결로 자동 폴백하는 **LLM-as-Judge** 루프를 구성했다. 모든 판단 호출에 temperature=0을 적용해 동일 입력에서 일관된 분기 결과를 보장했다.

### Agent 3 — 멘토 매칭 (`agents/mentor_match_agent.py`)

> 단순 직무 유사도 매칭이 아니다. 멘티가 어디서 막혔는지, 어떤 방향으로 전환하려는지, 어떤 역량을 이전할 수 있는지를 함께 반영해 멘티의 막힌 지점을 가장 잘 해결할 수 있는 멘토를 고른다. 멘티에게 "누구를 찾아야 할지"까지 요구하지 않는다.

**3채널 임베딩 통합** — 멘토의 프로필 텍스트, 커리어 전환 경험, 기존 답변 이력을 별도 채널로 임베딩해 통합 유사도를 산출한다. 프로필 텍스트만으로는 "이 멘토가 실제로 이런 질문에 잘 답하는가"를 측정할 수 없기 때문에 실제 답변 이력 채널을 별도로 추가했다.

**규칙 필터 + 재정렬** — 임베딩 유사도 이후 도메인 일치·병목 유형 일치·전환 경험 보유 여부를 보너스로 반영해 재정렬한다. 최종 점수는 유사도×0.55 + 멘토 만족도×0.20 + 활동량×0.10 + 수용가능성×0.15 가중합으로 산출한다.

**품질 평가 루프** — 후보 수가 3명 미만이거나 평균 점수가 0.40 미만이면 임계값을 완화(0.30 → 0.15)해 재시도한다. 최대 2회 시도 후 LLM이 Top-3를 최종 선별하고 각 멘토에 대한 추천 근거를 생성한다.

**프롬프트 엔지니어링 설계** — RERANK_PROMPT에 7개 평가 기준(job_match·career_path_match·domain_match·similar_question_exp·availability_fit·bottleneck_fit·transition_bridge_fit)을 상/중/하/정보부족/N/A 척도로 채점하도록 설계해 정성 판단을 구조화했다. 멘티 구조화 프로파일(source_role, current_bottleneck, bridge_hypothesis, needed_expertise 등)을 컨텍스트로 명시적으로 전달해(**contextual prompting**) LLM이 단순 직무명 비교가 아닌 병목 해결 가능성 중심으로 판단하도록 유도했다. 환각 방지를 위해 프롬프트에 "후보 목록에 있는 mentor_id 중에서만 선택"을 명시하고, 출력 후 valid_ids와 대조해 후보 외 ID를 제거하는 이중 검증을 적용했다. temperature=0.3으로 설정해 추천 근거 서술의 일관성을 확보했다.

---

### Agent 4 — 답변 자산화 (`agents/assetize.py`)

> 멘토의 1:1 답변을 다음 멘티가 재사용할 수 있는 자산으로 전환한다. 이 에이전트가 동작할수록 Agent 2의 직접 답변 비율이 높아지고 멘토에게는 개인 맥락이 중요한 질문만 전달된다 — 사용이 늘수록 서비스 품질과 운영 효율이 동시에 개선되는 자기강화 구조다.

**이중 동의 + 4기준 품질 판단** — 멘토·멘티 양측의 동의 확인 후, 개인 맥락 강도·정보 밀도·시의성·기밀성 4개 기준으로 LLM이 자산화 가능 여부를 판단한다. 탈락 시 `reject_reason`을 구조화해 기록하므로 어떤 유형의 답변이 자산화되지 못하는지 운영 중에 추적할 수 있다.

**시의성 3단계 처리** — Agent 2와 동일하게 stale / stale_but_useful / timeless 3단계로 구분한다. "오래됐지만 여전히 유용한" 답변을 단순 탈락 처리하면 초기 자산 손실이 크기 때문에 조건부 통과 등급을 별도로 설계했다.

**임베딩 + 메타데이터 태그 생성** — 통과한 답변은 임베딩 생성과 함께 도메인·직무·질문 유형·병목 유형·커리어 단계 태그를 LLM이 생성해 저장한다. 이 태그가 Agent 2의 검색 필터링 정확도에 직접 영향을 미친다.

**프롬프트 엔지니어링 설계** — 4가지 품질 판단(개인맥락강도·정보밀도·시의성·기밀성)을 하나의 프롬프트로 묶지 않고 독립적인 4개 LLM 호출로 분리했다. 판단 간 간섭을 없애고, 어떤 기준에서 탈락했는지 독립적으로 추적하기 위해서다. 각 프롬프트에 통과/실패 기준을 구체적인 사례와 함께 열거해(**criteria-based prompting**) 경계 케이스에서도 일관된 판단이 나오도록 했다(예: isolated 판단 기준 — "학교명·나이·지역이 3개 이상 조합되어 논리를 구성할 경우"). 메타데이터 생성(ASSET_METADATA_PROMPT)은 품질 판단과 완전히 분리된 별도 호출로 설계해 탈락 답변에 불필요한 토큰이 소모되지 않도록 했다. 전체 temperature=0으로 고정해 자산화 여부 판단의 재현성을 확보했다.

세션별 임시 상태와 사용자의 장기 프로필을 혼합하면 프로필이 오염된다는 문제를 방지하기 위해 DB를 두 계층으로 분리했다.

| 계층 | 파일 | 역할 |
|------|------|------|
| 장기 프로필 | `mentees.json` / `mentors.json` | 반복 사용 시 누적되는 멘티 맥락, 멘토 프로필 |
| 세션 단위 | `question_sessions.json` | 한 번의 질문 흐름에서 Agent 1이 생성한 구조화 출력 |

Agent 1이 세션을 종료할 때 `current_bottleneck`과 `taxonomy_tags`가 멘티 장기 프로필에 반영되어, 다음 세션에서 에이전트가 이전 맥락을 활용할 수 있다.

## 🔬 개발 과정에서의 검증

AI 출력물을 그대로 신뢰하지 않고, 어떤 판단이 안정적이고 어떤 판단이 어려운지 직접 확인했다.

### 사전 작업: 잇다 Q&A 50건 분석

평가 설계 전에 실제 잇다 멘토링 플랫폼의 Q&A 50건을 수집·분석했다. 각 질문을 직무 도메인, 질문 유형, 막힌 지점의 성격, 멘토에게 기대하는 답변 유형으로 분류하고, 이 분석 결과를 바탕으로 두 가지 산출물을 만들었다.

첫째, Agent 1의 `current_bottleneck` 9개 표준 라벨과 `hard_case_flags` 감지 규칙 — 실제 질문 패턴에서 귀납적으로 도출했다.

둘째, 초기 자산 DB — 50건을 직접 구조화해 `seed_real_cases.py`로 씨딩했다.

### Agent 1 평가 설계

50건 중 복합질문, 직무전환, 최신트렌드, 자료피드백 요청, 가능성 판단 등 판단이 까다로운 12개 케이스를 선별해 사람이 직접 gold annotation을 달았다. Gold JSON은 각 케이스마다 `question_units`, `current_bottleneck`, `hard_case_flags`, `routing_hints`, `taxonomy_tags`, `refined_question`의 기대 출력값을 포함한다.

`eval/eval_runner.py`가 Agent 1의 `analyze()` 메서드를 single-shot으로 호출하고, 출력 결과를 gold와 자동 비교한다. 각 평가 항목은 단순 일치/불일치가 아니라 의미적 overlap이나 LLM judge를 활용해 채점한다.

**평가 항목 및 가중치**

| 평가 항목 | 가중치 | 점수 | 채점 방식 | 해석 |
|----------|--------|------|----------|------|
| privacy_safe | 5% | **1.00** | PII 패턴 매칭 | 개인정보 제거 완벽 |
| hard_case_flags | 20% | **0.92** | 플래그 일치 비교 | 위험 신호 감지 안정적 |
| routing | 15% | 0.64 | 필드 완전 일치 | mentor_first 강하나 search_first 분기 개선 여지 |
| refined_question | 5% | 0.58 | GPT-4o-mini LLM judge | 방향은 맞지만 일부 세부 요소 누락 |
| current_bottleneck | 20% | 0.42 | 완전 일치 | 유사 병목 라벨 간 혼동 |
| taxonomy_tags | 10% | 0.26 | 태그 Recall | domain 잡지만 role·question_type 약함 |
| question_units | 25% | 0.29 | 키워드 overlap | 복합 질문을 하나로 압축하는 경향 |
| **전체 평균** | | **0.54** | | |

**결과 해석**

위험 감지(hard_case_flags 0.92)와 개인정보 보호(privacy_safe 1.00)는 안정적으로 작동한다. 복합 질문 분해(question_units 0.29)와 병목 라벨 분류(current_bottleneck 0.42)는 상대적으로 어려운 판단임을 확인했다. 이를 반영해 Agent 2·3은 Agent 1의 구조화 결과를 확정값으로 신뢰하기보다 각 단계에서 독립적으로 재판단하는 구조로 설계했다.

**Agent 2·3 평가 설계에 대하여**

Agent 2·3의 성능은 에이전트 판단 품질과 DB 커버리지가 함께 측정값에 영향을 미치는 구조여서, 초기 DB 규모에서 정량 지표만으로 에이전트 자체의 품질을 분리 측정하기 어렵다. 이 프로젝트에서는 설계 원칙과 판단 로직의 타당성 검증에 집중했으며, 실 데이터 누적 이후 정량 평가를 추가할 예정이다.

## 🔭 확장 방향

맨투맨의 구조는 데이터가 누적될수록 자기강화된다.

멘토 답변이 쌓일수록 Agent 2가 더 많은 질문을 기존 자산으로 처리할 수 있다. 반복되는 질문은 자산화 답변으로 처리하고, 개별 맥락이 강한 질문만 멘토에게 전달된다. 멘토 부담은 줄고, 멘티는 더 빠른 답변 경험을 얻는다.

**사용자 축:** 비수도권 대학생 → 비전공 직무 전환자 → 경력단절자 → 시니어 재취업자

**기관 축:** 대학 진로개발센터 → 지자체 청년정책 부서 → 고용노동부 청년진로 사업

확장이 가능한 이유는 맨투맨이 특정 직무 정보를 직접 제공하는 것이 아니라, **경험을 가진 사람과 그 경험이 필요한 사람 사이의 접근 경로 비대칭을 매개하는 구조 자체를 서비스**하기 때문이다.
